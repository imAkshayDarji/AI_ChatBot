## 1. Local Development Setup

### 1.1 Prerequisites

| Tool | Version | Purpose |
|---|---|---|
| Python | 3.12 | Backend runtime |
| Node.js | 20 LTS | Frontend runtime |
| pnpm | Latest | Frontend package manager |
| Docker | Latest | PostgreSQL + pgvector |
| Git | Latest | Version control |

### 1.2 Docker PostgreSQL

Local PostgreSQL with pgvector extension:

```bash
docker run -d \
  --name krystal-postgres \
  -e POSTGRES_USER=krystal \
  -e POSTGRES_PASSWORD=krystal_dev \
  -e POSTGRES_DB=krystal_studio \
  -p 5432:5432 \
  pgvector/pgvector:pg16
```

Verify the container is running:

```bash
docker ps | grep krystal-postgres
```

Connect to verify pgvector extension:

```bash
docker exec -it krystal-postgres psql -U krystal -d krystal_studio -c "CREATE EXTENSION IF NOT EXISTS vector;"
```

### 1.3 Backend Setup

```bash
cd apps/api

# Create virtual environment
python -m venv venv
source venv/bin/activate  # macOS/Linux

# Install dependencies
pip install -r requirements.txt

# Copy environment template
cp .env.example .env
# Edit .env with local values

# Run database migrations
alembic upgrade head

# Seed admin user
python -m scripts.seed_admin

# Start development server
uvicorn app.main:app --reload --port 8000
```

### 1.4 Frontend Setup

```bash
cd apps/web

# Install dependencies
pnpm install

# Copy environment template
cp .env.example .env.local
# Edit .env.local with local values

# Start development server
pnpm dev
```

### 1.5 Makefile Commands

The project includes a `Makefile` at the root for common operations:

| Command | Description |
|---|---|
| `make dev-api` | Start FastAPI backend with hot reload |
| `make dev-web` | Start Next.js frontend with hot reload |
| `make dev-db` | Start Docker PostgreSQL |
| `make test-api` | Run backend test suite |
| `make test-web` | Run frontend test suite |
| `make lint` | Run linting for both backend and frontend |
| `make migrate` | Run database migrations |
| `make migrate-down` | Rollback last migration |
| `make seed` | Seed admin user and sample knowledge |
| `make setup` | First-time setup (install deps, create DB, migrate, seed) |

## 2. Railway Backend Deployment

### 2.1 Railway Setup

1. Create a Railway account at [railway.app](https://railway.app).
2. Install the Railway CLI: `npm install -g @railway/cli`.
3. Login: `railway login`.
4. Link the project: `railway link` (select the backend service).

### 2.2 Railway PostgreSQL

1. Add a PostgreSQL service in the Railway project dashboard.
2. Railway auto-generates a `DATABASE_URL` environment variable.
3. Enable the pgvector extension after the database is created:

```bash
# Connect to Railway PostgreSQL shell
railway connect postgres

# Enable pgvector
CREATE EXTENSION IF NOT EXISTS vector;
```

### 2.3 Backend Service Configuration

**Dockerfile** (`apps/api/Dockerfile`):

```dockerfile
FROM python:3.12-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

EXPOSE 8000

CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

### 2.4 Railway Environment Variables

Set these in the Railway dashboard under the backend service:

| Variable | Example Value | Required |
|---|---|---|
| `DATABASE_URL` | Auto-generated by Railway PostgreSQL | Yes |
| `JWT_SECRET` | A cryptographically random string (32+ chars) | Yes |
| `JWT_ALGORITHM` | `HS256` | Yes |
| `ACCESS_TOKEN_EXPIRE_MINUTES` | `60` | Yes |
| `AI_PROVIDER` | `openai` | Yes |
| `OPENAI_API_KEY` | `sk-...` | Yes |
| `CHAT_MODEL` | `gpt-4o-mini` | Yes |
| `EMBEDDING_MODEL` | `text-embedding-3-large` | Yes |
| `CORS_ORIGINS` | `https://krystaltattoostudio.com` | Yes |
| `STUDIO_PHONE` | `+91-XXXX-XXXXXX` | Yes |
| `STUDIO_INSTAGRAM_URL` | `https://www.instagram.com/krystaltattoostudio` | Yes |
| `ENVIRONMENT` | `production` | Yes |
| `PORT` | `8000` | Auto |

### 2.5 Deployment Steps

```bash
# Deploy from CLI
railway up

# Or deploy via GitHub integration (recommended)
# Railway auto-deploys on push to main branch
```

### 2.6 Post-Deployment Verification

1. Check the health endpoint: `GET /api/v1/health` returns `{"status": "healthy"}`.
2. Check logs: `railway logs` for startup messages and errors.
3. Run migrations: `railway run alembic upgrade head`.
4. Seed admin user: `railway run python -m scripts.seed_admin`.
5. Test login: `POST /api/v1/admin/auth/login` with admin credentials.
6. Test chat: `POST /api/v1/chat/message` with a simple query.

## 3. Vercel Frontend Deployment

### 3.1 Vercel Setup

1. Create a Vercel account at [vercel.com](https://vercel.com).
2. Install the Vercel CLI: `npm install -g vercel`.
3. Login: `vercel login`.

### 3.2 Project Configuration

**`apps/web/vercel.json`** (if needed for custom configuration):

```json
{
  "framework": "nextjs",
  "buildCommand": "pnpm build",
  "outputDirectory": ".next"
}
```

### 3.3 Vercel Environment Variables

Set these in the Vercel dashboard under the project settings:

| Variable | Example Value | Required |
|---|---|---|
| `NEXT_PUBLIC_API_URL` | `https://your-railway-backend.up.railway.app` | Yes |
| `NEXT_PUBLIC_STUDIO_NAME` | `Krystal Tattoo Studio` | Yes |

**Important:** Never set backend secrets (`OPENAI_API_KEY`, `JWT_SECRET`, `DATABASE_URL`) in Vercel. Vercel environment variables are exposed to the browser if prefixed with `NEXT_PUBLIC_`.

### 3.4 Deployment Steps

```bash
cd apps/web

# Deploy to Vercel
vercel --prod
```

Or via GitHub integration: Vercel auto-deploys on push to `main` branch.

### 3.5 Production Domain

- Configure custom domain in Vercel dashboard.
- Update `CORS_ORIGINS` in Railway backend to match the production domain.
- Ensure SSL certificate is active (Vercel handles this automatically).

### 3.6 Preview Deployments

- Vercel creates preview deployments for every pull request.
- Preview deployments use the same `NEXT_PUBLIC_API_URL` as production.
- Useful for testing frontend changes before merging.

## 4. CI/CD with GitHub Actions

### 4.1 Workflow File

**`.github/workflows/ci.yml`:**

```yaml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  backend-lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
      - run: pip install flake8
      - run: cd apps/api && flake8 app/ --max-line-length=120 --exclude=migrations

  backend-tests:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: pgvector/pgvector:pg16
        env:
          POSTGRES_USER: test
          POSTGRES_PASSWORD: test
          POSTGRES_DB: test_db
        ports:
          - 5432:5432
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
    env:
      DATABASE_URL: postgresql+asyncpg://test:test@localhost:5432/test_db
      JWT_SECRET: test-jwt-secret
      AI_PROVIDER: openai
      OPENAI_API_KEY: sk-test-key
      CHAT_MODEL: gpt-4o-mini
      EMBEDDING_MODEL: text-embedding-3-large
      ENVIRONMENT: test
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
      - run: cd apps/api && pip install -r requirements.txt
      - run: cd apps/api && python -m pytest app/tests/ -v --tb=short

  frontend-lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: "20"
      - run: cd apps/web && pnpm install
      - run: cd apps/web && pnpm lint

  frontend-typecheck:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: "20"
      - run: cd apps/web && pnpm install
      - run: cd apps/web && pnpm typecheck
```

### 4.2 CI Pipeline

On every push and pull request to `main`:

```text
┌─────────────────┐
│  Backend Lint    │
├─────────────────┤
│  Backend Tests   │  ← Requires PostgreSQL service container
├─────────────────┤
│  Frontend Lint   │
├─────────────────┤
│  Frontend TC     │  ← Type checking
└─────────────────┘
```

### 4.3 CD Pipeline (Main Branch Only)

On push to `main` (after CI passes):

```text
┌──────────────────────┐
│ Railway Auto-Deploy  │  ← Backend
├──────────────────────┤
│ Vercel Auto-Deploy   │  ← Frontend
└──────────────────────┘
```

Both Railway and Vercel auto-deploy from the GitHub `main` branch. No manual deployment step required for routine changes.

### 4.4 GitHub Secrets

Store these secrets in the GitHub repository settings:

| Secret | Purpose |
|---|---|
| `RAILWAY_TOKEN` | Railway deployment token (if using CLI deploy) |
| `VERCEL_TOKEN` | Vercel deployment token (if using CLI deploy) |

## 5. Database Migrations in Production

### 5.1 Migration Process

```text
1. Create migration locally:
   cd apps/api && alembic revision --autogenerate -m "description"

2. Review the generated migration file:
   - Verify the UP migration does what you expect.
   - Verify the DOWN migration reverses it correctly.
   - Check for destructive operations (DROP, DELETE).

3. Test locally:
   alembic upgrade head
   alembic downgrade -1
   alembic upgrade head

4. Commit the migration file.

5. Deploy to production:
   railway run alembic upgrade head

6. Verify:
   - Check application logs for errors.
   - Test affected endpoints.
   - Verify health check passes.
```

### 5.2 Migration Rules

- **Never** run auto-destructive migrations without manual review.
- **Never** use `DROP TABLE`, `TRUNCATE`, or `DELETE all rows` in production.
- **Always** test migrations locally before running in production.
- **Always** ensure the DOWN migration works.
- **Always** backup the database before schema changes.

### 5.3 Migration Checklist

Before running a production migration:

- [ ] Migration file has been reviewed by a developer.
- [ ] Migration has been tested locally (both UP and DOWN).
- [ ] Database backup has been verified.
- [ ] Migration command is ready: `railway run alembic upgrade head`.
- [ ] Rollback plan is documented: `railway run alembic downgrade -1`.
- [ ] Post-migration verification steps are defined.

## 6. Production CORS Configuration

### 6.1 Setting CORS Origins

The `CORS_ORIGINS` environment variable in Railway must be set to the exact production frontend URL:

```text
CORS_ORIGINS=https://krystaltattoostudio.com,https://www.krystaltattoostudio.com
```

### 6.2 Updating CORS

If the production domain changes:

1. Update `CORS_ORIGINS` in Railway dashboard.
2. Railway redeploys the backend with the new configuration.
3. Verify the frontend can reach the backend.
4. Test chat and admin login.

### 6.3 CORS Debugging

If CORS errors occur:

- Check that the frontend URL exactly matches `CORS_ORIGINS` (including protocol and www).
- Check browser developer tools for the exact error.
- Verify `CORS_ORIGINS` in Railway logs.
- Test with `curl -H "Origin: https://your-domain.com" https://your-backend.railway.app/api/v1/health`.

## 7. Health Check Endpoints

### 7.1 Backend Health Check

`GET /api/v1/health`

Response:

```json
{
  "status": "healthy",
  "version": "1.0.0",
  "environment": "production"
}
```

Railway uses this endpoint for health monitoring. Configure it in the Railway service settings:

- Health check path: `/api/v1/health`
- Health check interval: 30 seconds
- Health check timeout: 10 seconds

### 7.2 Database Health

The health endpoint verifies database connectivity:

```python
@app.get("/api/v1/health")
async def health_check(db: AsyncSession = Depends(get_db)):
    try:
        await db.execute(text("SELECT 1"))
        db_status = "connected"
    except Exception:
        db_status = "disconnected"

    return {
        "status": "healthy" if db_status == "connected" else "degraded",
        "version": settings.APP_VERSION,
        "environment": settings.ENVIRONMENT,
        "database": db_status,
    }
```

### 7.3 Frontend Health

The frontend is considered healthy if:

- The Next.js build succeeds.
- The page loads without errors.
- The API URL is correctly configured.

Vercel monitors this automatically.

## 8. Backup Strategy

### 8.1 Railway PostgreSQL Backups

- Railway provides automated daily backups for PostgreSQL.
- Backup retention: 7 days (configurable in Railway dashboard).
- Backups include all data, including pgvector embeddings.

### 8.2 Backup Verification

Monthly backup verification:

1. Download the latest backup from Railway dashboard.
2. Restore to a local PostgreSQL instance.
3. Verify all tables and data are intact.
4. Verify pgvector extension works with restored data.
5. Test the application against the restored database.

### 8.3 Manual Backup Before Migrations

Before running any production database migration:

1. Trigger a manual backup in the Railway dashboard.
2. Wait for the backup to complete.
3. Verify the backup exists in the backup list.
4. Proceed with the migration.

### 8.4 Backup Restoration

To restore from a backup:

1. Contact Railway support or use the Railway dashboard.
2. Select the backup to restore.
3. Wait for restoration to complete.
4. Verify the application connects to the restored database.
5. Test critical endpoints (health, login, chat).
